Service `boards_service.archive_board` was the `NotImplementedError` stub from PR 137 — now implemented end-to-end. Idempotency choice: **preserve original timestamp** rather than refresh on every call, because the stamp records *when* the board first left active circulation and overwriting loses audit value (the issue body said "200 no-op on repeat" — preserve-timestamp is the strict reading). Repository surface untouched (reuses `board_repo.get_by_id` + direct attribute mutation + flush — no new repo method needed since archive is a single-attribute write that doesn't justify a `board_repo.archive` wrapper). Same `MissingGreenlet` trap as `update_board`: after `flush`, `TimestampMixin.updated_at` (`onupdate=func.now()`) is DB-computed and SQLAlchemy expires the attribute; without `await session.refresh(board, ["updated_at"])` before commit, the router's `BoardRead.model_validate(board)` triggers a lazy-load outside async context. Fix is identical to PR 142's. Archive does **not** cascade — columns / labels / tasks survive on the underlying tables; the archived board is still reachable via `GET /api/boards/{id}?include_archived=true` with relations intact (locked by a dedicated pytest case). Seven new pytest cases in `tests/test_boards_archive.py` cover first-call-200, idempotent-preserves-timestamp, default-list-excludes, include-archived-shows, 404, 401, and the cascade-safety invariant. Mandatory Playwright spec extended at `frontend/tests/e2e/api/boards.spec.ts` from 7 to 8 scenarios — picks up the deferred `TODO(#73)` from PR 140 and adds the full archive flow the original issue 70 body called for (POST 2 boards → archive one → default list excludes → `?include_archived=true` includes → repeat archive returns same `archived_at`).
