The single repo call `board_repo.get_by_id_with_relations` uses `selectinload(Board.columns)` + `selectinload(Board.labels)` to keep the load to exactly 3 SELECTs against the `boards` / `columns` / `labels` tables regardless of N. `joinedload` was deliberately avoided — the cartesian product of columns x labels would re-emit board / column rows once per label, blowing payload transfer. Column ordering by `position` is inherited from the relationship's `order_by="Column.position"` declared on the model (PR 137); no extra `order_by` needed in the loader. Service-level archive policy mirrors `list_boards`: archived boards 404 by default, `?include_archived=true` returns them — same status code (404, not 410 / 403) so unauthorized probers can't tell archived-but-exists from never-existed. Seven new pytest cases in `tests/test_boards_get.py` cover empty relations, position ordering with reverse-order inserts, unknown id 404, archived-default 404, archived-with-include-archived 200, no-auth 401, and the load-bearing N+1 assertion (5 columns + 5 labels → ≤3 SELECTs against the three relevant tables; auth-middleware queries on `users` / `refresh_tokens` are excluded so the assertion stays on the eager-load shape, not on unrelated middleware churn). Mandatory Playwright spec extended at `frontend/tests/e2e/api/boards.spec.ts` from 4 to 6 scenarios — `GET /api/boards/{id}` with auth → 200 with `columns: []` + `labels: []` (after creating a fresh board), and `GET /api/boards/9999999` → 404 (with auth, since unauthenticated would 401 before reaching the not-found path).
