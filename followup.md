Issue 80 added `POST /api/boards/{board_id}/columns/reorder` — transactional batch position rewrite. Endpoint uses the **sub-resource shape** (mirrors POST `/boards/{board_id}/columns` create — the operation is inherently board-scoped, so the URL pins the scope rather than parsing it from the body). New `ColumnsReorderRequest` schema with `ordered_column_ids: list[int]` (`min_length=1` blocks empty at 422). Service `reorder_columns` validates set-equality against the board's existing columns AND list-length-vs-set-length parity (catches duplicates independent of board contents — `[A, A, B]` against board `{A, B}` would multiset-match the perfect case but the explicit length check guarantees rejection). Four invalid input shapes — partial / extra / duplicate / unknown — collapse onto one `ColumnError("invalid_reorder")` → 400 because the client UX is the same in every case ("the list you sent doesn't match the board"). Same archived-board → 404 obscurity contract as PATCH/DELETE. Position rewrite uses `COLUMN_POSITION_STEP * (i + 1)` (1000/2000/3000/...), full rewrite rather than ADR-0004's float `(prev + next) / 2` — column reorders are rare AND the typical column count is 3-7, so cascading integer rewrites stay cheap; the float scheme exists for the *task* hot path. `MissingGreenlet` trap fix per `update_column`: explicit `await session.refresh(col, ["updated_at"])` after flush before commit. 10 pytest cases lock: 200 reverse, 200 idempotent same-order, 400 partial, 400 extra-id-from-other-board (cross-board-leak guard), 400 duplicate, 400 unknown id, 404 unknown board, 404 archived board with no-mutation assertion, 422 empty list, 401 no-auth. Mandatory Playwright spec extends `frontend/tests/e2e/api/columns.spec.ts` (now 8 scenarios — was 6): reorder reverses 3 columns + GET parent board reflects new order via `BoardDetailRead.columns` (eager-loaded with `order_by=Column.position`); partial-list 400 + GET confirms positions unchanged. Module docstring updated — reorder is no longer "coming with #80".
