Issue 79 added `DELETE /api/columns/{column_id}` — protected delete that refuses on a non-empty column. Endpoint uses the **flat shape** `/columns/{column_id}` (id at root, board resolved from DB), matching #78. Service raises `ColumnError("column_has_tasks")` with `task_count` attached as an instance attribute; router branches on `exc.code` and hand-builds a `JSONResponse` for the 409 because FastAPI's `HTTPException(detail={...})` would nest the structure as `{"detail": {"detail": "...", "task_count": N}}` rather than the issue-spec flat `{"detail": "column_has_tasks", "task_count": N}`. Same archived-board → 404 obscurity contract as PATCH (`BoardError("board_not_found")`). Repo gains `task_count_for_column(column_id)` (single `SELECT COUNT(*)`, hits the `(column_id, position)` index) and `delete(column)`. `Column.tasks` already declares `cascade="all, delete-orphan"` and the FK is `ON DELETE CASCADE` so a naive delete would silently take every task with it — the 409 guard exists precisely to surface that as a deliberate user decision. 8 pytest cases lock: 204 happy path, 204 sibling-safety (defensive — a widened delete would still 204 to caller), two 409 task-count echoes (1 task, 5 tasks), 409 no-mutation, unknown-id 404, archived-board 404, no-auth 401. Tasks seeded directly via `Task(...)` + `session.add()` because public POST /api/tasks lands with #90. Mandatory Playwright spec extends `frontend/tests/e2e/api/columns.spec.ts` (now 6 scenarios — was 4): DELETE empty column → 204 + GET parent board no longer lists it (cross-endpoint contract, with sibling-survives check); DELETE unknown id → 404. The 409 smoke scenario is **deferred** with a `TODO(#90)` breadcrumb at the top of the spec — pytest covers 409 end-to-end, the gap is HTTP-level cross-endpoint and reclaims itself once #90 ships.
