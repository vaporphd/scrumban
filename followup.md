Service `boards_service.update_board` was the `NotImplementedError` stub from PR 137 — now implemented end-to-end. Repository gains `board_repo.apply_updates` which mutates attrs + flushes (rather than emitting a separate `update().where().values()` Core statement) because the row is already loaded for 404 + archive enforcement, so a roundtrip-saving Core update would not actually save a roundtrip — and `TimestampMixin.updated_at` (`onupdate=func.now()`) refreshes the same way under both paths. Service uses `payload.model_dump(exclude_unset=True)` not `exclude_none`: clients that explicitly send `description: null` get the field cleared (write NULL), clients that omit the field leave it alone — `exclude_none` would conflate the two and silently turn clear-requests into no-ops. Empty body `{}` is a documented no-op (no schema-level "at least one field required" rule — returns the current row unchanged with 200). Archive policy mirrors `get_board`: archived boards 404 (not 403, not 410) so probers can't tell archived-but-exists from never-existed; same model means PATCH on an archived board is indistinguishable from PATCH on a missing one. **Critical SQLAlchemy gotcha caught**: after `apply_updates` flushes with `onupdate=func.now()`, SQLAlchemy expires `updated_at` (DB computed it, ORM doesn't know the new value); without `await session.refresh(board, ["updated_at"])` before commit, the router's `BoardRead.model_validate(board)` triggers a lazy-load outside async context → `MissingGreenlet`. Fixed in service. Ten new pytest cases in `tests/test_boards_update.py` cover each-field-only, both-at-once, null-clears, no-op, two 422 shapes (empty / oversize), unknown-id 404, archived 404 with no-mutation assertion, and 401. Mandatory Playwright spec extended at `frontend/tests/e2e/api/boards.spec.ts` from 6 to 7 scenarios — POST a board → PATCH the name → re-GET → assert new name visible AND description preserved (proves persistence + partial-update semantics in one scenario).
