# 04 — Сравнение «облако vs локальная» + рекомендация

## Таблица сравнения

Сопоставляем облачного ассистента (Claude Sonnet/Opus 4.x из предыдущих дней) с
лучшей локальной связкой — **Qwen3.6-35B-A3B MLX @ 4bit** на 36GB Mac.

| Критерий | Облако (Claude 4.x) | Локальная (Qwen3.6-35B-A3B MLX 4bit) |
|---|---|---|
| **Качество кода** | ⭐⭐⭐⭐⭐ Идиоматика React/TS из коробки | ⭐⭐⭐⭐ Хорошо для задач «внутри файла», иногда теряет детали между файлами |
| **Кросс-файловое рассуждение** | ⭐⭐⭐⭐⭐ Держит 10+ файлов в голове | ⭐⭐⭐ Надёжно до ~5 файлов, дальше начинает забывать |
| **Скорость первого ответа** | 2–4 сек (сетевой round-trip) | 4–6 сек при 30k ctx; растёт до 10–15 сек к 60k |
| **Скорость генерации** | ~60–80 tok/s (облако) | ~22–28 tok/s (локальная) |
| **Понимание контекста проекта** | Отлично, особенно с CLAUDE.md/AGENTS.md | Хорошо, если контекст ≤60k; дальше — Truncate отрезает нужное |
| **Работа без интернета** | ❌ Нет | ✅ Да (при работающем LAN до сервера) |
| **Приватность кода** | Зависит от политики провайдера | Полная — ничего не уходит за пределы LAN |
| **Свежие API / актуальные доки** | ✅ Да (WebSearch, обновлённые данные) | ❌ Только то, что в обучающем датасете |
| **Tool use / агентный режим** | ✅ Стабильно, много шагов | 🟡 Работает, но деградирует после 4–6 шагов |
| **Стоимость** | Деньги за токены | Амортизация железа + электричество |
| **Лимиты** | Rate limits, дневные потолки | Ограничение только пропускной способностью сервера |
| **Стабильность** | Практически 100% | OOM при плохой настройке; зацикливания на плохих квантах |

## Для каких задач локальная модель достаточна

✅ **Точно хватает локальной:**

- Генерация отдельных React-компонентов и хуков по описанию.
- Написание тестов для уже существующего кода.
- Локальный рефакторинг внутри файла (переименование, выделение функций).
- Заполнение boilerplate — конфиги, типы, props.
- Объяснение/разбор существующего кода.
- Commit-сообщения, docstring-и, README-фрагменты.
- Автокомплит в процессе набора кода.

🟡 **Работает, но с контролем:**

- Агентные задачи на 3–5 шагов (правь файл → запусти тесты → почини).
- Рефакторинг, затрагивающий 2–4 файла с чётким ТЗ.
- Чат про чужую библиотеку, если код и типы есть в проекте.

❌ **Незаменимо облако:**

- Кросс-репо рефакторинг, затрагивающий ≥10 файлов.
- Отладка ошибок в свежих версиях библиотек (Vite 6, React 19, TS 5.5+).
- Поиск решения через веб (StackOverflow, GitHub issues).
- Работа с редкими API, edge-cases, нестандартными стеками.
- Сложное архитектурное планирование с множеством trade-offs.
- Длинные диалоги >100k токенов (локально просто не влезает в VRAM).
- Ответственные задачи, где цена ошибки высока (security, migrations, payment).

## Лучшая связка на нашем железе

**Hardware:** Mac 36GB (сервер) + Mac 24GB (клиент, IDE).

**Стек:**

| Слой | Выбор |
|---|---|
| Сервер | LM Studio 0.4.12 (Metal/MLX backend) |
| Модель | Qwen3.6-35B-A3B MLX @ 4bit |
| Контекст | 64k (`loaded_context_length`), Overflow → Rolling Window |
| Семплинг | Temp 0.4 / Top K 40 / Top P 0.9 / Min P 0.05 / Repeat 1.1 |
| System-память | `iogpu.wired_limit_mb = 28000` |
| Клиент IDE | OpenCode (чат/автокомплит) + Codex CLI (агент) |
| System prompt | `AGENTS.md` проекта |

**Почему именно эта связка:**

1. MoE 35B-A3B выигрывает у dense 9B по качеству при сопоставимом TTFT на
   префилле. 3B активных параметров → prefill в 2–3 раза быстрее 9B dense.
2. 4bit — единственный квант, который стабильно влезает в 36 GB вместе с
   64k KV-кэшем без OOM.
3. 64k контекста покрывает 95% рабочих диалогов; редкие длинные сессии
   всё равно выгоднее переносить в облако.
4. OpenCode + Codex используют один и тот же endpoint, поэтому конфиг
   модели/промпта не дублируется, и переключение «чат ↔ агент» бесшовное.

## Итоговая рекомендация

> **Локальной хватает на 70–80% повседневных кодинг-задач внутри знакомого
> проекта при наличии 36+ GB unified memory. Остальные 20–30% (кросс-репо,
> свежие API, длинный контекст, критичная ответственность) — через облако.**

Практическая стратегия:

1. **Дефолт — локальная.** Включили LM Studio, работаем в OpenCode.
2. **Эскалация в облако** — когда заметили, что модель теряет детали между
   файлами, или когда задача требует поиска по интернету / свежих доков.
3. **Жёсткая эскалация** — ответственные операции (миграции БД, security
   review, переписывание payments) → сразу в облако.

Такой workflow на нашем железе даёт:

- Приватность для 70–80% кода.
- Независимость от сети для рутины.
- Скорость — без очередей/rate limits.
- Качество и глубину рассуждения — там, где это действительно нужно.
